\section{Motivation}
\subsection{Why This Problem Matters}
\begin{frame}[t]{Motivation: Why This Problem Matters}
\scriptsize
\begin{itemize}
  \setlength{\itemsep}{0.25em}
  \item In distributed storage reliability, two mainstream choices are replication and erasure coding; erasure coding saves substantial storage, and RS codes have been widely deployed with strong theory and system optimizations.
  \item Two-tone \sxor has excellent theoretical complexity: it uses shift/XOR computations instead of finite-field matrix multiplications.
  \item But prior Two-tone studies mainly emphasize coding theory, with limited system-level optimization for memory access behavior.
  \item RS-oriented optimization techniques cannot be directly transferred to \sxor codes because both computation principles and data-access patterns differ.
  \item We therefore need a dedicated implementation methodology for Two-tone \sxor and practical validation in production-like systems such as Ceph.
\end{itemize}

\vspace{0.3em}
\begin{block}{Talk Goal}
Build system-level, memory-access-aware optimizations for \twotone \sxor and verify real performance gains in Ceph.
\end{block}
\end{frame}
\note{
In distributed storage, reliability is usually provided by replication or erasure coding. Replication is simple, while erasure coding is much more storage-efficient. In practice, Reed-Solomon is already widely used and has strong theoretical and system-level optimizations, so our work is not about dismissing RS. Our focus is Two-tone Shift-XOR, which has excellent theoretical complexity because it replaces finite-field matrix multiplications with shifts and XORs. The gap is implementation: prior Two-tone work provides limited system-level optimization for memory access. RS-specific optimization tricks cannot be directly reused because the computation principles and data-access patterns are different. So our goal is to build Shift-XOR-specific implementation optimizations and validate them in Ceph.
}

\section{Limitations of Existing Solutions}
\subsection{Encoding-Side Limitations}
\begin{frame}[t]{Limitations of Existing Solutions (1/2)}
\scriptsize
\begin{columns}[T,totalwidth=\textwidth]
  \column{0.44\textwidth}
  \begin{itemize}
    \item \textbf{Redundant Memory Access and Poor Temporal Locality:} Naive Two-tone encoding/decoding repeatedly updates parity locations without memory-friendly scheduling.
    \item \textbf{Inefficient Data Traversal Pattern of Encoding:} Horizontal traversal has better data locality but repeated parity updates; vertical traversal reduces parity updates but increases data reloads.
  \end{itemize}

  \column{0.56\textwidth}
  \centering
  \includegraphics[width=\linewidth,height=0.56\textheight,keepaspectratio]{Example_of_Two-tone_Encoding.pdf}

  \vspace{0.2em}
  {\scriptsize Fig.1: Example of Two-tone encoding in the paper.}
\end{columns}
\end{frame}
\note{
This slide covers the first two limitations from the paper. First, Redundant Memory Access and Poor Temporal Locality: naive Two-tone encoding and decoding repeatedly touch parity memory without memory-friendly scheduling. Second, Inefficient Data Traversal Pattern of Encoding: horizontal traversal has better data locality but repeated parity updates, while vertical traversal reduces parity updates but increases data reloads. So the issue is not coding correctness, but memory-access trade-offs in implementation. Fig.1 provides the encoding context.
}

\subsection{Decoding-Side Limitations}
\begin{frame}[t]{Limitations of Existing Solutions (2/2)}
\scriptsize
\begin{columns}[T,totalwidth=\textwidth]
  \column{0.43\textwidth}
  \begin{itemize}
    \item \textbf{Complex Decoding Process with Suboptimal Traversal Pattern}
    \item Naive decoding treats data recovery and parity re-encoding as separate operators.
    \item For mixed data/parity erasures, this causes repeated passes over overlapping memory regions.
    \item This motivates operator fusion and case-aware traversal selection.
  \end{itemize}

  \vspace{0.4em}
  \textbf{Takeaway:} the bottleneck is memory behavior in implementation, not the Two-tone coding theory itself.

  \column{0.57\textwidth}
  \centering
  \includegraphics[width=\linewidth,height=0.56\textheight,keepaspectratio]{Example_of_Two-tone_Decoding.pdf}

  \vspace{0.2em}
  {\scriptsize Fig.2: Example of Two-tone decoding in the paper.}
\end{columns}
\end{frame}
\note{
The third limitation is Complex Decoding Process with Suboptimal Traversal Pattern. Naive decoding often separates data recovery and parity re-encoding into two operators. For mixed data and parity erasures, this creates repeated passes over overlapping memory regions. This is exactly where operator fusion and case-aware traversal can help. Fig.2 shows the decoding context.
}

\section{Approach and Key Ideas}
\subsection{Optimization Overview}
\begin{frame}[t]{FastTT Overview}
\scriptsize
\begin{columns}[T,totalwidth=\textwidth]
  \column{0.44\textwidth}
  \begin{block}{Core design principle}
  Optimize \twotone with memory-access-aware system techniques, not just algorithmic structure.
  \end{block}

  \vspace{0.2em}
  \begin{enumerate}
    \item Cache-Friendly Window Partitioning
    \item Neighbor-Merging Access
    \item Erasure-Parity Mapping
    \item Traversal Pattern Selection
  \end{enumerate}

  \vspace{0.2em}
  \begin{itemize}
    \item Reduce redundant memory traffic
    \item Improve locality across encoding and decoding
    \item Specialize the decoder for common erasure cases
  \end{itemize}

  \column{0.56\textwidth}
  \centering
  \includegraphics[width=\linewidth]{Two-tone_Optimization_Overview.pdf}

  \vspace{0.2em}
  {\scriptsize Overview of the FastTT workflow, adapted from the paper.}
\end{columns}
\end{frame}
\note{
Our solution is FastTT, and the key design principle is to optimize Two-tone from a system perspective, especially memory access behavior. FastTT combines four techniques: cache-friendly window partitioning, neighbor-merging access, erasure-parity mapping, and traversal pattern selection. Together, these techniques reduce redundant memory traffic, improve locality, and adapt the decoder to common failure patterns. This is the core story of the talk.
}

\subsection{Cache-Friendly Window Partitioning}
\begin{frame}[t]{Key Idea 1: Cache-Friendly Window Partitioning}
\scriptsize
\begin{columns}[T,totalwidth=\textwidth]
  \column{0.43\textwidth}
  \begin{itemize}
    \item Treat each chunk with a windowed, tiling-style processing strategy.
    \item Apply all \sxor updates inside one window before moving to the next window.
    \item Do not initialize the whole chunk to zero in advance; write values directly during shift-XOR updates.
    \item This removes one full initialization traversal and lowers memory-access overhead.
  \end{itemize}

  \vspace{0.4em}
  \textbf{Effect:} simpler access scheduling and less unnecessary memory traffic.

  \column{0.57\textwidth}
  \centering
  \includegraphics[width=\linewidth,height=0.54\textheight,keepaspectratio]{Cache-Friendly_Window_Demonstration.pdf}

  \vspace{0.2em}
  {\scriptsize Windowed processing plays a tiling-like role, improving locality without relying on hardware-specific cache sizing.}
\end{columns}
\end{frame}
\note{
The first idea is a windowed, tiling-style processing strategy. Instead of scanning an entire chunk at once, we work window by window and complete the shift-XOR updates inside one window before moving to the next. The important implementation detail is that we do not pre-initialize the whole chunk to zero. We simply write the values directly during shift-XOR updates, which removes one full traversal and reduces memory-access overhead. The main benefit is a simpler access schedule and less unnecessary memory traffic.
}

\subsection{Traversal Patterns and Neighbor-Merging Access}
\begin{frame}[t]{Key Idea 2: Traversal Patterns and Neighbor-Merging Access}
\scriptsize
\begin{columns}[T,totalwidth=\textwidth]
  \column{0.41\textwidth}
  \begin{itemize}
    \setlength{\itemsep}{0.35em}
    \item Horizontal traversal ($\Theta((k+km)l)$): scans data chunk-by-chunk, giving strong local reads but many repeated parity writes.
    \item Vertical traversal ($\Theta((m+km)l)$): writes each parity symbol once, but repeatedly reloads data from different chunks.
    \item Neighbor-merging (FastTT, $\Theta\bigl((k+\tfrac{km}{2})l\bigr)$): XORs adjacent data contributions in SIMD registers before touching parity memory, cutting write traffic while preserving locality.
  \end{itemize}

  \vspace{0.3em}
  \textbf{Design choice:} keep the horizontal-style access locality, but reduce parity-memory pressure with in-register merging.

  \column{0.59\textwidth}
  \centering
  \begin{minipage}[t]{0.48\linewidth}
    \centering
    \includegraphics[width=\linewidth,height=0.145\textheight,keepaspectratio]{Two-tone_Horizontal_Access_Pattern.pdf}

    {\tiny (a) Horizontal traversal}
  \end{minipage}\hfill
  \begin{minipage}[t]{0.48\linewidth}
    \centering
    \includegraphics[width=\linewidth,height=0.145\textheight,keepaspectratio]{Two-tone_Vertical_Access_Pattern.pdf}

    {\tiny (b) Vertical traversal}
  \end{minipage}

  \vspace{0.45em}

  \begin{minipage}[t]{0.96\linewidth}
    \centering
    \includegraphics[width=\linewidth,height=0.205\textheight,keepaspectratio]{Neighbor-Merging_Access_Pattern.pdf}

    {\tiny (c) Neighbor-merging access in FastTT}
  \end{minipage}
\end{columns}
\end{frame}
\note{
Two-tone encoding trades between two main traversal patterns with distinct memory-access behaviors. Horizontal traversal ($\Theta((k+km)l)$) scans data chunk-by-chunk and updates parity symbols per-step: this yields very good local reads but causes many parity writes. Vertical traversal ($\Theta((m+km)l)$) gathers all data symbols needed for a parity symbol and writes once: it reduces parity writes but repeatedly reloads data. FastTT's neighbor-merging reduces extra memory traffic by combining adjacent contributions inside SIMD registers before writing parity, lowering the effective cost to $\Theta\bigl(k+\tfrac{km}{2}\bigr)l$ and improving performance on memory-bound platforms. This vertical pattern will come back later, because FastTT reuses it as a special fast path for single-erasure decoding.
}

\subsection{Erasure-Parity Mapping}
\begin{frame}[t]{Key Idea 3: Erasure-Parity Mapping}
\scriptsize
\begin{columns}[T,totalwidth=\textwidth]
  \column{0.42\textwidth}
  \begin{itemize}
    \item Traditional decoding often does two steps: recover erased data, then re-encode erased parity.
    \item FastTT maps erased data and erased parity into one unified abstraction.
    \item Decoded codewords are immediately reused to update erased parity inside the same elimination flow.
    \item This removes a separate repair stage and avoids extra traversal.
  \end{itemize}

  \vspace{0.4em}
  \textbf{Benefit:} fewer memory round-trips, no separate repair pass, and less redundant reading and writing.

  \column{0.58\textwidth}
  \centering
  \includegraphics[width=\linewidth,height=0.54\textheight,keepaspectratio]{Erasure-Parity_Mapping_Decoding_Demonstration.pdf}

  \vspace{0.2em}
  {\scriptsize Dashed arrows and crossed-out steps show work eliminated by integrating parity recovery into decoding.}
\end{columns}
\end{frame}
\note{
The third idea is to unify erased data and erased parity under one mapping abstraction. Traditionally, the decoder first reconstructs lost data and then runs a separate parity repair phase. FastTT removes that split. Instead, one decoding workflow handles both kinds of losses together, and decoded codewords are immediately reused to update erased parity inside the elimination process. So the key point is not that every case follows exactly the same path, but that we avoid writing results out and reading them back again for a separate repair pass. This cuts redundant memory traffic and makes the decoding workflow cleaner.
}

\subsection{Traversal Pattern Selection}
\begin{frame}[t]{Key Idea 4: Traversal Pattern Selection}
\scriptsize
\begin{columns}[T,totalwidth=\textwidth]
  \column{0.42\textwidth}
  \begin{itemize}
    \item In real clusters, a single erased chunk is much more common than multi-chunk loss.
    \item For that special case, running full \twotone elimination is unnecessary.
    \item FastTT reuses the vertical pattern introduced earlier and selects it only for one-erasure recovery.
    \item For all other cases, it falls back to the unified standard decoder rather than trying to optimize every pattern.
  \end{itemize}

  \vspace{0.4em}
  \textbf{Result:} lower control overhead, a one-pass recovery path, and very large gains for single erasures.

  \column{0.58\textwidth}
  \centering
  \includegraphics[width=\linewidth,height=0.54\textheight,keepaspectratio]{Vertical_Traversal_Pattern_Demonstration.pdf}

  \vspace{0.2em}
  {\scriptsize A missing chunk is reconstructed in one pass over survived chunks, without full two-tone elimination.}
\end{columns}
\end{frame}
\note{
The fourth idea is based on a practical observation: in real storage systems, single-erasure recovery is much more common than complicated multi-erasure recovery. This is where the earlier vertical traversal pattern comes back. FastTT is not doing a fully general smart selection for every loss pattern. Instead, when exactly one chunk is erased, it explicitly chooses the vertical pattern and reconstructs the missing chunk with a direct one-pass XOR path. For all other cases, it falls back to the unified standard decoder. That narrower fast path is enough to cut control overhead and is a major reason why single-erasure decoding speed improves so dramatically.
}

\section{Ongoing/Future Research \& Summary}
\subsection{Summary and Future Research}
\begin{frame}[t]{Summary and Future Research}
\small
\begin{block}{One-sentence summary}
\design is the first high-performance systems implementation of \twotone \sxor coding, turning a theoretically efficient code family into a practical high-throughput storage primitive.
\end{block}

\vspace{0.5em}
\textbf{What to remember}
\begin{itemize}
  \item The main bottleneck is memory behavior, not just arithmetic complexity.
  \item Four lightweight but coordinated optimizations unlock large gains in both encoding and decoding.
  \item The performance foundation is to improve locality together with reducing memory-access count; the single-erasure case is then a special opportunity for extra decoding speedup.
\end{itemize}

\vspace{0.4em}
\textbf{Open directions}
\begin{itemize}
  \item Adaptive or heterogeneous window sizing
  \item More flexible SIMD-aware merging heuristics, e.g., multi-level pairwise merging
  \item Further improving decoding efficiency for cases with more erased chunks
\end{itemize}
\end{frame}
\note{
To summarize in one sentence, FastTT is the first high-performance systems implementation of Two-tone Shift-XOR coding. The key lesson is that the bottleneck is memory behavior, so the real performance gains come from coordinating multiple methods that both improve locality and reduce total memory-access count. On top of that, the single-erasure case gives an extra opportunity for a specialized fast decoding path. Looking forward, there are several promising directions: adaptive window sizing, more flexible SIMD-aware merging heuristics such as multi-level pairwise merging, and continuing to improve decoding efficiency when more chunks are erased. We hope this work makes Shift-XOR coding a more practical option for real storage systems.
}
