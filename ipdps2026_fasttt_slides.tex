\begin{frame}[t]{Motivation: Why This Problem Matters}
\small
\begin{itemize}
  \item In distributed storage reliability, two mainstream choices are replication and erasure coding; erasure coding saves substantial storage, and RS codes have been widely deployed with strong theory and system optimizations.
  \item Two-tone \sxor has excellent theoretical complexity: it uses shift/XOR computations instead of finite-field matrix multiplications.
  \item But prior Two-tone studies mainly emphasize coding theory, with limited system-level optimization for memory access behavior.
  \item RS-oriented optimization techniques cannot be directly transferred to \sxor codes because both computation principles and data-access patterns differ.
  \item We therefore need a dedicated implementation methodology for Two-tone \sxor and practical validation in production-like systems such as Ceph.
\end{itemize}

\vspace{0.6em}
\begin{block}{Talk Goal}
Build system-level, memory-access-aware optimizations for \twotone \sxor and verify real performance gains in Ceph.
\end{block}
\end{frame}

\begin{frame}[t]{Limitations of Existing Solutions (1/2)}
\scriptsize
\begin{columns}[T,totalwidth=\textwidth]
  \column{0.44\textwidth}
  \begin{itemize}
    \item \textbf{Redundant Memory Access and Poor Temporal Locality:} Naive Two-tone encoding/decoding repeatedly updates parity locations without memory-friendly scheduling.
    \item \textbf{Inefficient Data Traversal Pattern of Encoding:} Horizontal traversal has better data locality but repeated parity updates; vertical traversal reduces parity updates but increases data reloads.
  \end{itemize}

\begin{frame}[t]{Limitations of Existing Solutions (2/2)}
\scriptsize
\begin{columns}[T,totalwidth=\textwidth]
  \column{0.43\textwidth}
  \begin{itemize}
    \item \textbf{Complex Decoding Process with Suboptimal Traversal Pattern}
    \item Naive decoding treats data recovery and parity re-encoding as separate operators.
    \item For mixed data/parity erasures, this causes repeated passes over overlapping memory regions.
    \item This motivates operator fusion and case-aware traversal selection.
  \end{itemize}

  \vspace{0.4em}
  \textbf{Takeaway:} the bottleneck is memory behavior in implementation, not the Two-tone coding theory itself.

\begin{frame}[t]{Key Idea 1: Cache-Friendly Window Partitioning}
\scriptsize
\begin{columns}[T,totalwidth=\textwidth]
  \column{0.43\textwidth}
  \begin{itemize}
    \item Treat each chunk with a windowed, tiling-style processing strategy.
    \item Apply all \sxor updates inside one window before moving to the next window.
    \item Do not initialize the whole chunk to zero in advance; write values directly during shift-XOR updates.
    \item This removes one full initialization traversal and lowers memory-access overhead.
  \end{itemize}

  \vspace{0.4em}
  \textbf{Effect:} simpler access scheduling and less unnecessary memory traffic.

  \column{0.57\textwidth}
  \centering
  \includegraphics[width=\linewidth,height=0.54\textheight,keepaspectratio]{Cache-Friendly_Window_Demonstration.pdf}

  \vspace{0.2em}
  {\scriptsize Windowed processing plays a tiling-like role, improving locality without relying on hardware-specific cache sizing.}
\end{columns}
\end{frame}

\begin{frame}[t]{Key Idea 2: Traversal Patterns and Neighbor-Merging Access}
\scriptsize
\begin{columns}[T,totalwidth=\textwidth]
  \column{0.41\textwidth}
  \begin{itemize}
    \setlength{\itemsep}{0.35em}
    \item Horizontal traversal ($\Theta((k+km)l)$): scans data chunk-by-chunk, giving strong local reads but many repeated parity writes.
    \item Vertical traversal ($\Theta((m+km)l)$): writes each parity symbol once, but repeatedly reloads data from different chunks.
    \item Neighbor-merging (FastTT, $\Theta\bigl((k+\tfrac{km}{2})l\bigr)$): XORs adjacent data contributions in SIMD registers before touching parity memory, cutting write traffic while preserving locality.
  \end{itemize}

  \vspace{0.3em}
  \textbf{Design choice:} keep the horizontal-style access locality, but reduce parity-memory pressure with in-register merging.

\begin{frame}[t]{Key Idea 3: Erasure-Parity Mapping}
\scriptsize
\begin{columns}[T,totalwidth=\textwidth]
  \column{0.42\textwidth}
  \begin{itemize}
    \item Traditional decoding often does two steps: recover erased data, then re-encode erased parity.
    \item FastTT maps erased data and erased parity into one unified abstraction.
    \item Decoded codewords are immediately reused to update erased parity inside the same elimination flow.
    \item This removes a separate repair stage and avoids extra traversal.
  \end{itemize}

  \vspace{0.4em}
  \textbf{Benefit:} fewer memory round-trips, no separate repair pass, and less redundant reading and writing.

  \column{0.58\textwidth}
  \centering
  \includegraphics[width=\linewidth,height=0.54\textheight,keepaspectratio]{Erasure-Parity_Mapping_Decoding_Demonstration.pdf}

  \vspace{0.2em}
  {\scriptsize Dashed arrows and crossed-out steps show work eliminated by integrating parity recovery into decoding.}
\end{columns}
\end{frame}

\begin{frame}[t]{Key Idea 4: Traversal Pattern Selection}
\scriptsize
\begin{columns}[T,totalwidth=\textwidth]
  \column{0.42\textwidth}
  \begin{itemize}
    \item In real clusters, a single erased chunk is much more common than multi-chunk loss.
    \item For that special case, running full \twotone elimination is unnecessary.
    \item FastTT reuses the vertical pattern introduced earlier and selects it only for one-erasure recovery.
    \item For all other cases, it falls back to the unified standard decoder rather than trying to optimize every pattern.
  \end{itemize}

  \vspace{0.4em}
  \textbf{Result:} lower control overhead, a one-pass recovery path, and very large gains for single erasures.

  \column{0.58\textwidth}
  \centering

  \vspace{0.2em}
  {\scriptsize A missing chunk is reconstructed in one pass over survived chunks, without full two-tone elimination.}
\end{columns}
\end{frame}

\begin{frame}[t]{Summary and Future Research}
\small
\begin{block}{One-sentence summary}
\design is the first high-performance systems implementation of \twotone \sxor coding, turning a theoretically efficient code family into a practical high-throughput storage primitive.
\end{block}

\vspace{0.5em}
\textbf{What to remember}
\begin{itemize}
  \item The main bottleneck is memory behavior, not just arithmetic complexity.
  \item Four lightweight but coordinated optimizations unlock large gains in both encoding and decoding.
  \item The biggest wins come from improving locality and specializing the dominant single-erasure case.
\end{itemize}

\vspace{0.4em}
\textbf{Open directions}
\begin{itemize}
  \item Adaptive or heterogeneous window sizing
  \item More flexible SIMD-aware merging heuristics
  \item Reusing the fast path for reducible multi-erasure patterns
\end{itemize}
